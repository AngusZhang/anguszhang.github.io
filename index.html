Posted on 2018-09-19 Edited on 2021-01-18 In JAVA

  • Hashtable，HashMap 与 TreeMap

    • HashTable：早期哈希表实现，使用 synchronized 来保证线程安全，因此性能开销较大，不支持 null 作为键或值。

    • HashMap：线程不安全，支持 null 作为键与值。结构为数组作为桶存储键值的 hash 相同的链表的头部节点，get、put、remove 时间复杂度为可达常数级别。性能表现非常依赖于哈希码的有效性，请务必掌握 hashCode 和 equals 的一些基本约定，比如：

      • equals 相等，hashCode 一定要相等。
      • 重写了 hashCode 也要重写 equals。
      • hashCode 需要保持一致性，状态改变返回的哈希值仍然要一致。
      • equals 的对称、反射、传递等特性。

    • TreeMap：基于红黑树实现，get、put、remove 时间复杂度为 O（log(n)），有序，根据指定的 Comparator 或键的自然顺序来判断顺序。

  • HashMap 的源码分析

    1. 内部实现
      我们知道 HashMap 的结构为数组加链表。数组即为 Node<K, V>[] table数组，相同 hash 值得键值对一链表形式存储在相同链表内，当链表大小超过阈值 TREEIFY_THRESHOLD 时，链表将会被改造为树形结构。
      • 门限值等于（负载因子）x（容量），如果构建 HashMap 的时候没有指定它们，那么就是依据相应的默认常量值。
      • 门限通常是以倍数进行调整 （newThr = oldThr << 1），我前面提到，根据 putVal 中的逻辑，当元素个数超过门限大小时，则调整 Map 大小。
      • 扩容后，需要将老的数组中的元素重新放置到新的数组，这是扩容的一个主要开销来源。
    2. 容量与负载系数
      为什么需要在乎容量与负载因子？这是因为容量和负载系数决定了可用的桶的数量，空桶太多会浪费空间，如果使用的太满则会严重影响操作的性能。极端情况下，假设只有一个桶，那么它就退化成了链表，完全不能提供所谓常数时间存的性能。

  • 解决 hash 冲突的方法

    • 开放定址法
      基本思想是：当关键字key的哈希地址 p=H(key) 出现冲突时，以p为基础，产生另一个哈希地址p1，如果p1仍然冲突，再以p为基础，产生另一个哈希地址p2，…，直到找出一个不冲突的哈希地址pi，将相应元素存入其中。
    • 再哈希法
      这种方法是同时构造多个不同的哈希函数：Hi = RH1(key); i=1，2，…，k。当哈希地址Hi=RH1（key）发生冲突时，再计算Hi=RH2（key）……，直到冲突不再产生。这种方法不易产生聚集，但增加了计算时间。
    • 链地址法
      这种方法的基本思想是将所有哈希地址为i的元素构成一个称为同义词链的单链表，并将单链表的头指针存在哈希表的第i个单元中，因而查找、插入和删除主要在同义词链中进行。链地址法适用于经常进行插入和删除的情况。
    • 建立公共溢出区
      这种方法的基本思想是：将哈希表分为基本表和溢出表两部分，凡是和基本表发生冲突的元素，一律填入溢出表。

Java核心技术36讲_02

Posted on 2018-07-12 Edited on 2021-01-18 In Java

  • 问题提出

    请对比 Exception 和 Error，另外，运行时异常与一般异常有什么区别？

  • 问题分析

    这个问题主要考察对 Exception 与 Error 的理解，只要可以阐述清楚就好。
    对此主要需要掌握两个方面。

    • 理解 Throwable、Exception、Error 的设计和分类

      下方是一张简单的类图，可供参考

      此外还有一些子类型需要重点理解一下，例如 ClassNotFoundException 和 NoClassDefFoundError 有什么区别？

      1. 二者一个是 Exception，一个是 Error，Exception 需要捕获，
而 Error 是不需要捕获的
2. 二者发生的原因与阶段也不相同。NoClassDefFoundError 表示该类
在编译阶段还可以找到，但是在运行Java应用的时候找不到了。
而 ClassNotFoundException 是在通过利用反射加载类的时候产生的。
3. ClassNotFoundException 是在代码中显示的使用Class.forName()等方法
加载类是抛出，原因通常是由于缺少了某些jar包导致。
而 NoClassDefFoundError 由 JVM 在连接阶段抛出，最有可能的原因
是jar包发生了冲突。

  • 异常处理的两个基本原则

    1. 第一，尽量不要捕获类似 Exception 这样的通用异常，而是应该捕获特定异常。这样做可以让代码更为清晰的表达一些信息，另外也可以避免捕获到一些不希望或者不应该捕获的异常。
    2. 不要生吞（swallow）异常，生吞异常会掩盖问题。
    3. Throw early, catch late。Throw early，这样做的好处可以在我们查找异常栈的时候可以更快地定位到问题发生的原因。Catch late，如果当前代码捕获后的异常不知道如何处理，可以重新组装后重新抛出，因为更上一层的逻辑会更清晰的知道异常需要如何处理。

  • 性能方面

    1. try-catch 会产生额外的性能开销，所以在建议只捕获需要捕获的代码段。
    2. 不要使用异常来控制代码流程，这是低效的。
    3. Java 每实例化一个 Exception 都会进行快照，可能会导致服务反应变慢、吞吐量下降。
